# AIX 环境安装与烧录提示词

> 用途：把本文档中“给伙伴 Codex 的提示词”整段复制给伙伴的 Codex。它会先检查仓库和环境，再安装依赖、运行测试、构建 ESP32-S3 固件，最后在确认串口和得到明确授权后烧录。
>
> 当前仓库根目录：`D:\Projects\IOTCompetition\ProjectFile`

## 一、仓库结构分析

```text
ProjectFile/
├─ AIX/                         ESP32-S3 ESP-IDF 固件

│  ├─ CMakeLists.txt            ESP-IDF 工程入口

│  ├─ dependencies.lock         ESP-IDF 5.4.4 与组件锁定版本
│  ├─ sdkconfig.defaults        默认安全配置；气动控制和自动模式均关闭
│  └─ sync_runtime_config.ps1   生成本机 Wi-Fi、token、设备 ID 和 PC 服务地址
├─ host_app/                    PySide6 上位机
│  ├─ aix_host_app/             UI、串口解析、HTTP 链路、会话记录
│  ├─ tests/                    Python 单元测试
│  ├─ requirements.txt          PySide6、pyserial、pyqtgraph、numpy
│  └─ start_host_app.cmd        推荐启动入口
├─ Models/DepthAnything3/       PC 视觉模型和 FastAPI 服务的唯一运行根目录

│  ├─ run_service.ps1           单独启动模型服务
│  ├─ source/                   安装后生成，未纳入 Git
│  ├─ env/                      安装后生成，Python 3.10 模型环境
│  ├─ weights/                  DA3-SMALL、YOLO26m 权重
│  ├─ cache/                    Hugging Face、Torch、Ultralytics 等缓存
│  └─ logs/                     模型服务日志
├─ scripts/
│  ├─ verify.ps1                主机测试、C 测试、compileall，可选固件全量构建
│  └─ run_e2e_acceptance.ps1    端到端验收辅助脚本
└─ docs/                        接线、设计、验收和本提示词
```

## 二、依赖矩阵

| 层级 | 必要环境或库 | 用途 |
| --- | --- | --- |
| 基础 | Windows PowerShell、Git | 执行脚本和管理仓库 |
| 主机 C 测试 | MinGW-w64 `gcc` | `scripts/verify.ps1` 编译 AIX 的主机侧 C 测试 |
| ESP32 固件 | ESP-IDF **5.4.4**、CMake、Ninja、ESP32-S3 Xtensa 工具链 | 配置、编译和烧录 `AIX` |
| 上位机 | Python 3.10–3.13、项目根 `.venv` | 运行 PySide6 上位机 |
| 上位机 Python 库 | `host_app/requirements.txt` | PySide6、串口、图表和数组 |
| PC 视觉服务 | 独立的 `Models/DepthAnything3/env` Python 3.10 环境 | 避免模型依赖污染上位机环境 |
| PC 视觉库 | CUDA PyTorch `2.12.1+cu126`、`xformers==0.0.35`、DA3、`addict`、FastAPI、Uvicorn、Ultralytics `8.4.96` 等 | DA3-SMALL + YOLO26m 推理 |
| 模型文件 | DA3-SMALL 和 YOLO26m 权重 | 真实模型服务启动和预热 |
| 硬件 | ESP32-S3-DevKitC-1、USB 数据线、正确的 COM 端口 | 固件烧录和串口监视 |

注意：`Models/DepthAnything3/install.ps1` 里的上游 DA3 运行依赖中显式补装了 `addict`，不能只依赖上游项目声明的 requirements。TensorRT FP16 引擎不是烧录固件的必要条件；如果导出失败，服务应回退到 CUDA FP16 的 `yolo26m.pt`，但不能静默回退到 CPU。

## 三、给伙伴 Codex 的提示词

将下面内容完整复制给伙伴 Codex：

```text
你现在负责在 Windows 上准备并验证这个 AIX 项目，目标是：安装必要环境和库，完成主机测试与 ESP32-S3 固件构建；在我明确确认串口后，再把已验证的固件烧录到板子并提供串口日志。

项目必须以当前工作树的 Git 根目录为准，不要假设固定盘符。先执行：

  $repo = (& git rev-parse --show-toplevel).Trim()
  Set-Location $repo
  git status --short --branch
  git log -1 --oneline

如果当前目录不是 Git 根目录，先定位真正包含 .git 的 ProjectFile 目录。不要从旧的 origin/main 重新取代码覆盖当前工作树；先确认当前提交包含 AIX、host_app、Models/DepthAnything3 和 docs 中的最新文件。如果远程仓库缺少当前工作树的提交，停止烧录并报告需要使用哪个分支/提交，不能把旧固件烧到板上。

重要工作规则：
1. 使用中文报告过程和结果。
2. 先做只读检查，再安装和构建；保留用户已有修改，不执行 git reset --hard、git checkout --、递归删除或清空虚拟环境。
3. 不要打印 Wi-Fi 密码、link token 或任何密钥；日志中只报告“已配置/未配置”。
4. 可以自动安装软件包、创建项目内虚拟环境、下载模型权重和生成 build 目录。
5. 真正 flash 前必须先列出检测到的串口、板卡和将要使用的构建目录，并向我确认；只有收到明确的“可以烧录/确认烧录”后才执行 flash。
6. 默认保持 CONFIG_AIX_ENABLE_PNEUMATIC_CONTROL=n 和 CONFIG_AIX_ENABLE_PNEUMATIC_AUTOMATIC=n。不要为了通过测试开启气泵、电磁阀或自动充气。
7. 构建通过不等于硬件验收通过；最终报告必须分开写“源码/测试/构建/烧录/实机日志”。

一、基础工具检查

检查以下命令是否存在并记录实际路径：

  git --version
  python --version
  cmake --version
  ninja --version
  gcc --version

如果缺少 Git、Python、MinGW-w64 gcc、CMake 或 Ninja，安装 Windows 对应版本。gcc 必须能在当前 PowerShell 中执行，因为 scripts/verify.ps1 会编译主机侧 C 测试。

二、准备上位机环境

只使用项目根目录的一个环境：

  $hostPython = Join-Path $repo '.venv\Scripts\python.exe'

如果该文件不存在，优先使用 Python 3.10、3.11 或 3.12 创建；若机器只有兼容的 Python 3.13，也可以使用它：

  py -3.10 -m venv .venv

如果 py -3.10 不存在，选择机器上可用的兼容 Python 创建 `.venv`，不要创建 `host_app\.venv`。

安装并验证上位机依赖：

  & $hostPython -m pip install --upgrade pip
  & $hostPython -m pip install -r (Join-Path $repo 'host_app\requirements.txt')
  & $hostPython -c "import PySide6, serial, pyqtgraph, numpy; print('host dependencies ok')"

三、准备 DA3/YOLO 模型环境

模型环境必须位于：

  $repo\Models\DepthAnything3\env

模型源代码、权重、缓存和日志也必须留在 `Models\DepthAnything3` 内，不要散落到用户目录或其他项目。先检查 `Models\DepthAnything3\install.ps1` 中的 Conda 路径：

  $conda = Get-Command conda -ErrorAction SilentlyContinue

如果脚本内写死的 Conda 路径不存在，不能直接失败后声称模型环境已安装。请做一个最小、可审查的适配：让脚本优先使用环境变量 `AIX_CONDA_EXE`，其次使用 `Get-Command conda` 找到的 conda.exe，最后才回退到脚本原来的路径；只修改环境路径兼容性，不改模型版本和推理逻辑。若本机没有 Conda，安装 Miniconda/Anaconda 后再继续。

执行模型安装脚本：

  Set-Location (Join-Path $repo 'Models\DepthAnything3')
  powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\install.ps1

安装过程中必须核对这些内容：
- Python 模型环境为 3.10；
- torch 为 CUDA 版本，`torch.cuda.is_available()` 为 True 时才标记 GPU 可用；
- 已安装 `xformers==0.0.35`、`addict`、FastAPI、Uvicorn、`ultralytics==8.4.96`；
- DA3-SMALL 和 YOLO26m 权重存在；
- 已生成 `Models\DepthAnything3\install_manifest.json`；
- TensorRT 引擎若不可用可以记录为未生成，但不能把 CPU 当作成功的视觉运行后端。

在导入 Ultralytics 或启动服务前，把缓存目录指向项目内，避免写入受限的用户目录：

  $modelRoot = Join-Path $repo 'Models\DepthAnything3'
  $env:DA3_ROOT = $modelRoot
  $env:HF_HOME = Join-Path $modelRoot 'cache\huggingface'
  $env:HF_HUB_CACHE = Join-Path $env:HF_HOME 'hub'
  $env:TORCH_HOME = Join-Path $modelRoot 'cache\torch'
  $env:YOLO_CONFIG_DIR = Join-Path $modelRoot 'cache\ultralytics'
  $env:MPLCONFIGDIR = Join-Path $modelRoot 'cache\matplotlib'
  New-Item -ItemType Directory -Force -Path $env:HF_HOME,$env:TORCH_HOME,$env:YOLO_CONFIG_DIR,$env:MPLCONFIGDIR | Out-Null

使用模型环境做最小检查：

  $modelPython = Join-Path $modelRoot 'env\python.exe'
  & $modelPython -c "import torch, fastapi, uvicorn, addict, ultralytics; print(torch.__version__); print('cuda_available=', torch.cuda.is_available())"

四、准备 ESP-IDF 5.4.4

AIX/dependencies.lock 已锁定 ESP-IDF 5.4.4，目标为 esp32s3。不要用 Arduino、PlatformIO 或其他框架替代它。

优先寻找已安装的 ESP-IDF：

  Get-ChildItem D:\APP\ESPIDF, C:\Espressif -Filter export.ps1 -Recurse -ErrorAction SilentlyContinue | Select-Object -First 10 FullName

在找到的 ESP-IDF 目录中运行其 `export.ps1`，然后在同一个 PowerShell 中检查：

  $env:IDF_PATH
  idf.py --version
  cmake --version
  ninja --version
  xtensa-esp32s3-elf-gcc --version

版本必须与锁定配置兼容，首选 ESP-IDF v5.4.4。若工具链命令只在父 PowerShell 可见、但 Python/CMake 子进程不可见，先修复 ESP-IDF shell 环境；不要只凭父进程的 `Get-Command` 判断成功。必要时使用 ESP-IDF 自带的原生 PowerShell，而不是普通 Codex shell。

五、生成本机运行时配置

运行时配置包含 Wi-Fi 凭据和 link token，不能提交 Git。若仓库没有可用的本机配置，使用现有交互式脚本录入 Wi-Fi：

  Set-Location $repo
  powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\AIX\configure_preview.ps1
  powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\AIX\sync_runtime_config.ps1

`configure_preview.ps1` 的名称是历史兼容名称；当前默认配置仍应由 `sync_runtime_config.ps1` 生成，保持视觉上行开启、相机预览关闭，并保持气动控制和自动模式关闭。若使用的不是 Windows Mobile Hotspot 的 `192.168.137.1`，按实际 PC 地址调用 `sync_runtime_config.ps1` 的 `-ServiceUrl` 参数。

检查 `AIX\sdkconfig.runtime`：
- 文件为 UTF-8 无 BOM；
- SSID 非空，密码不在终端输出；
- 有设备 ID 和随机 link token；
- PC 服务地址正确；
- 不包含开启气动自动模式的配置。

六、运行仓库验证

先运行主机、模型和 C 测试：

  Set-Location $repo
  powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\scripts\verify.ps1

然后在已激活 ESP-IDF 5.4.4 的同一个 PowerShell 中运行完整固件构建：

  powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\scripts\verify.ps1 -BuildFirmware

必须检查这些产物：
- `AIX\build-verify\AIX.bin` 存在；
- `AIX\build-verify\sdkconfig` 已生成，并核对其实际配置而不是只看旧的 sdkconfig；
- `AIX\build-verify\firmware-manifest.json` 存在；
- `git diff --check` 通过。

如果报 `Ninja`、`cmake` 或 `xtensa-esp32s3-elf-gcc` 找不到，先回到 ESP-IDF 原生 shell 修复 PATH，再重跑构建；不要修改业务代码来绕过环境错误。

七、启动 PC 服务和上位机（可选，但建议在烧录前做）

确保 Windows 2.4 GHz Mobile Hotspot 已准备好，然后执行：

  Set-Location (Join-Path $repo 'host_app')
  .\start_host_app.cmd

启动器会启动 8008 端口的模型服务，等待 `/healthz`，再启动 PySide6 上位机。先确认 `GET http://127.0.0.1:8008/healthz` 的 `http_ready=true`，且模型后端不是 CPU。关闭上位机时也要确认后台模型服务已退出。

八、烧录前检查和烧录

先不要烧录。列出串口并确认 USB 数据线连接的是目标 ESP32-S3：

  Get-CimInstance Win32_SerialPort | Select-Object DeviceID,Caption,Description

不要默认使用 COM21，也不要杀掉未知进程抢串口。如果端口被占用，先关闭串口监视器、Arduino、PlatformIO 或其他已知占用者；无法判断占用者时停下来报告。

向我报告以下信息并等待明确确认：
- 当前 Git 提交短哈希；
- `AIX\build-verify\AIX.bin` 的完整路径和 SHA256；
- ESP-IDF 版本；
- 检测到的 COM 端口及设备描述；
- `sdkconfig` 中气动控制和自动模式仍为关闭；
- 是否已完成相机/板卡接线检查。

收到明确“可以烧录”后，使用刚才验证过的 `build-verify` 构建目录，不要重新编译成另一份未知固件：

  Set-Location (Join-Path $repo 'AIX')
  idf.py -B build-verify -p COMxx flash monitor

把 `COMxx` 替换为实际确认的端口。串口监视器退出使用 Ctrl-]。烧录后至少收集并报告：启动日志、Wi-Fi 连接状态、设备 ID、camera_status、action_status；如果连接了 DFPlayer，再报告 `voice_status`。不要把“烧录成功”写成“整机安全验收成功”。

九、最终报告格式

最后用以下五项汇报，明确写出 PASS、FAIL 或 BLOCKED：

1. 环境：Python、Conda、ESP-IDF、CMake、Ninja、gcc 的版本和路径；
2. 依赖：上位机依赖、DA3/YOLO 依赖、权重和 GPU 状态；
3. 自动化验证：`verify.ps1`、`verify.ps1 -BuildFirmware`、`git diff --check` 的结果；
4. 烧录：是否获得授权、使用的 COM 端口、固件 SHA256、idf.py 结果；
5. 实机：实际串口日志和仍未验证的项目边界。

任何安装失败、模型下载失败、GPU 不可用、ESP-IDF 版本不符、构建失败、串口不明确或没有烧录授权，都要保留现场证据并停止在对应阶段，不能用“理论上可以”替代结果。
```

## 四、烧录后的边界

本项目默认固件只保证软件链路和安全默认值：气泵、电磁阀及自动充气保持关闭。即使上位机测试、模型服务测试和 ESP-IDF 构建全部通过，也不能替代 OV5640、MPU6050、DFPlayer、气路、泵、阀、电池、MOSFET 和气囊的实物验收。实物验收应继续依据：

- [ESP32-S3 固件说明](../AIX/README.md)
- [上位机说明](../host_app/README.md)
- [OV5640 接线说明](hardware/ov5640-devkitc1-wiring.md)
- [DFPlayer 语音接线与验收](hardware/dfplayer-voice-wiring.md)
- [气泵、三通电磁阀与 MPU6050 接线说明](hardware/pneumatic-mpu6050-wiring.md)

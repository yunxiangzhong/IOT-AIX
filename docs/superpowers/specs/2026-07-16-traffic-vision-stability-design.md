# 交通视觉识别与上位机稳定化设计

## 目标

在 RTX 4060 Laptop 8GB、i7-13700H 上，将现有 `OV5640 -> ESP32-S3 -> PC -> ESP32-S3` 闭环改造成稳定的 1 帧/秒交通视觉链路。模型完成预热后，目标整链路延迟为 P95 不超过 1000ms；上位机只展示已完成分析的帧，并保证画面、检测框、风险结果和动作确认使用同一 `frame_seq`。

## 模型与推理

- 以官方 COCO 预训练 `YOLO26m` 替换当前 SSDLite，检测 `person/bicycle/car/motorcycle/bus/truck/traffic light/stop sign`。默认使用本机导出的 TensorRT FP16 引擎，若引擎不可用则回退到 CUDA FP16 `.pt`，不得回退到 CPU 静默运行。
- 保留 DA3-SMALL 提供相对深度，运行分辨率固定为 280。服务启动时用 QVGA 合成 JPEG 完成 DA3 与 YOLO 多轮预热，预热完成前 `/healthz.model_ready=false`。
- 模型源码、依赖环境、缓存、权重和 TensorRT 引擎全部保留在 `Models/DepthAnything3` 下；安装清单记录 YOLO 文件、版本、哈希和实际执行后端。

## 稳定风险与快照

- 风险改为“交通目标优先”：全场景相对深度只能贡献至注意级，不能在没有交通目标时直接产生高危/严重；移除按类别名计算框增长的逻辑，避免同类多目标互相覆盖制造尖峰。
- 连续分数采用非对称 EMA（上升 0.45、下降 0.20）；等级使用滞回与确认状态机。普通升级需连续 2 帧，降级需连续 3 帧；仅当高置信度、中央、极近目标满足紧急条件时允许单帧升至严重。发布分数必须落在发布等级对应区间，继续兼容 ESP32 的分数/等级一致性校验。
- PC 服务保留“最新上传原图”，另维护“最新已处理快照”。只有分析完成后才原子提交快照、检测框和稳定风险；新增 `/v1/frame/processed.jpg`，`chain_state.display` 指向同一快照。
- `action_ack` 增加 `e2e_latency_ms`，由 ESP32 使用同一启动时钟计算“采集到接受风险”的完整延迟；PC 状态和上位机显示该指标。

## 上位机行为

- 主画面改为固定尺寸策略的自绘画布，不再向 `QLabel` 反复设置不同 `QPixmap`。画布采用等比留黑边，并用 QPainter 叠加中文类别、置信度和目标风险框。
- 客户端按 `chain_state.display.frame_seq` 拉取已处理帧，并把当时的状态副本随图像一起发送给仪表盘；仪表盘在一次调用中更新图像、框和风险卡。
- `chain_state` 是风险卡的唯一数据源；串口 `action_status` 只更新动作反馈区，且忽略旧帧，不再覆盖风险分数/等级。
- 状态轮询保留，但相同 revision 不重复重绘或写诊断日志。画面默认约 1 秒更新一次，状态文案改为“已分析画面”，不再宣称实时。

## 固件与时序

- `CONFIG_AIX_VISION_UPLOAD_PERIOD_MS` 默认改为 1000ms；相机仍可按 200ms 采集，上传任务只保留最新合格帧，不积压历史帧。
- 风险 TTL 继续保持 3000ms 作为失效保护；单次网络波动可以丢帧，但不得把旧帧延迟回传为新结果。
- 预热完成后的 PC 分析目标 P95 <= 500ms，真实板端整链路目标 P95 <= 1000ms。冷启动预热时间单独展示，不计入运行期帧延迟。

## 验收

- 自动测试覆盖：场景风险上限、同类多目标、单帧尖峰、2 帧升级/3 帧降级、紧急升级、快照原子性、processed endpoint、旧帧丢弃、串口不覆盖风险、固定画布 size hint、1000ms 上传配置和 E2E ACK。
- 本机模型基准至少 30 个热帧，报告平均值、P95、最大值和显存；不以官方 T4 数据替代本机数据。
- 上位机离屏渲染 1280x720 与 1440x900，人工检查画面区域不改变尺寸、检测框对齐、中文标签可读、风险卡不被旧串口事件覆盖。

## 明确边界

- 本次使用通用 COCO 交通类别，不训练自定义交通数据集；后续可用真实骑行数据微调。
- 风险分数是相对视觉风险，不是碰撞概率；自动气动执行仍受现有固件开关、标定和安全策略约束。
- 真实 Wi-Fi/板端 P95 需要烧录后实机采样；若当前没有可用串口设备，交付时必须明确标记该项未实测，不能用 PC 回环冒充。

# 撞击级加速度突变、保护性充气与上位机求助设计

## 背景

当前 MPU6050 运动策略同时包含“连续两次绝对加速度模长不低于 2.0 g”和快速倾斜判断。远端提交还把危险锁存的自动清除条件放宽为只检查加速度模长和角速度，导致设备横向倒地并静止 5 秒后也会清除危险状态。

本次改动把“撞击”统一定义为短时间内加速度模长发生巨大变化。普通起步、刹车、转弯、路面颠簸和姿态变化不触发保护动作。只有撞击级加速度突变才进入 MPU 碰撞保护分支：请求气囊充气、向本地上位机显示持续求助告警并记录事件。该分支不触发 DFPlayer。

## 目标

1. 删除旧的“连续两次绝对加速度模长不低于 2.0 g”撞击判据。
2. 使用唯一撞击判据：相邻有效样本的加速度模长变化不低于 1.2 g，且采样间隔大于 0 ms、不超过 20 ms。
3. 撞击后锁存危险；设备横向倒地时不得自动清除锁存。
4. 撞击与现有视觉 `HIGH/CRITICAL` 风险并列成为自动气动保护来源；快速倾斜只保留为运动诊断，不再单独触发气囊。
5. 上位机弹出非模态、持续显示的碰撞求助告警，并写入独立事件记录。
6. 气动是否允许执行必须由固件和上位机使用一致的自检、压力与自动模式条件。

## 非目标

- 不向外部云平台、短信、电话或第三方服务发送 SOS。
- 不为碰撞事件播放 DFPlayer 语音。
- 不改变视觉 `HIGH/CRITICAL` 风险的既有自动气动路径。
- 不取消压力、超压、充气超时、急停、泵阀自检或固件自动模式开关。
- 本次只做源码、测试和构建验证，不刷写开发板；实物气囊是否正常动作仍需单独授权后验证。

## 方案选择

### 采用方案：相邻样本模长差

在 100 Hz MPU6050 数据流中计算：

```text
accel_delta_g = abs(current_accel_norm_g - previous_accel_norm_g)
impact_event = 0 < sample_interval_ms <= 20
               and accel_delta_g >= 1.2
               and not in 200 ms refractory period
```

加速度模长差不会把单纯改变佩戴方向当成撞击；20 ms 时间窗覆盖正常 10 ms 采样及一次轻微调度抖动。超过 20 ms 的样本只更新基线，不参与撞击判断，避免串口、I2C 或任务阻塞后把跨越较长时间的变化误判为瞬时碰撞。首次撞击事件后的 200 ms 内继续刷新基线但不重复产生事件，用来把同一次物理撞击的陡升、陡降和传感器振铃合并为一次；首次事件仍立即触发保护。

### 未采用方案

- `g/s` jerk：数学表达更直接，但对毫秒时间戳抖动和极小时间间隔更敏感。
- 滑动窗口峰峰值：抗噪更强，但会增加检测延迟和状态复杂度，不适合当前 100 Hz 原型的最小改动目标。

## 固件设计

### 运动检测状态

`motion_detector_t` 增加上一有效加速度模长和时间戳。校准完成时用最后一个校准样本初始化基线；校准前不产生撞击事件。

`motion_output_t` 保留现有 `impact` 字段作为兼容协议字段和气动策略输入，其语义仍为“撞击危险已锁存”，但锁存的唯一来源改为新的加速度突变判据。新增：

- `accel_delta_g`：当前样本与上一有效样本的模长差；
- `sample_interval_ms`：参与本次判断的样本间隔；
- `impact_event`：仅在本次采样首次检出撞击级突变时为真；
- `impact_count`：本次设备启动后检出的累计撞击次数，使用无符号计数器；
- `impact`：撞击危险是否仍处于锁存状态；
- `danger_latched`：撞击或快速倾斜诊断是否仍处于锁存状态。

旧的 `MOTION_DETECTOR_IMPACT_THRESHOLD_G`、`MOTION_DETECTOR_IMPACT_SAMPLES` 和连续样本计数器全部删除，替换为：

```c
#define MOTION_DETECTOR_IMPACT_DELTA_G 1.2f
#define MOTION_DETECTOR_IMPACT_MAX_INTERVAL_MS 20ULL
#define MOTION_DETECTOR_IMPACT_REFRACTORY_MS 200ULL
```

### 危险锁存清除

撞击发生时立即锁存 `impact_latched` 并清零稳定计时。只有以下条件连续满足 5 秒才允许自动清除：

- 相对校准重力方向的倾角小于 30°；
- 加速度模长位于 0.8–1.2 g；
- 角速度模长小于 20°/s。

横向倒地、倒置或持续运动均不会清除锁存。新的撞击或快速倾斜会重新开始稳定计时。

### 气动触发

`pneumatic_policy_input_t` 继续使用 `motion_impact`，但其语义变为新的撞击级突变锁存。自动触发优先级保持：

1. 视觉 `CRITICAL`；
2. 视觉 `HIGH`；
3. MPU 撞击级加速度突变。

`motion_rapid_tilt` 不再进入 `current_automatic_trigger()`，避免普通快速姿态变化触发气囊；它仍通过运动遥测提供诊断。

气动执行前必须同时满足：

- 固件编译时启用气动控制和自动模式；
- 压力样本有效且新鲜；
- 泵自检通过；
- 阀自检通过；
- 自检没有失败锁存；
- 标定配置有效。

碰撞发生但条件不满足时，固件不得绕过保护门控。上位机仍显示碰撞求助，并明确显示“气囊未执行”及阻塞原因。

控制任务在同一互斥锁内快照泵验证、阀验证、自检失败和待处理命令，再调用策略，避免 HTTP 命令任务与控制任务之间读取到不一致的安全状态。

### 压力与时长

保留现有软件限制：默认目标压力 8 kPa、默认硬上限 12 kPa、可配置硬上限不超过 20 kPa、最大充气时长 5000 ms。上位机初始值与固件保持一致。所有保护性充气仍受超压、压力无效、压力过期、充气超时、保压超时和急停逻辑控制。

## 协议设计

运动串口消息继续兼容锁存语义的 `impact`，并增加可选诊断字段：

```json
{
  "type": "motion",
  "version": 2,
  "accel_norm_g": 2.31,
  "accel_delta_g": 1.31,
  "sample_interval_ms": 10,
  "impact_event": true,
  "impact_count": 1,
  "impact": true,
  "rapid_tilt": false,
  "danger_latched": true
}
```

上位机解析器对新字段提供默认值，以便继续读取未升级固件的 version 2 消息。为避免 100 Hz 检测事件落在现有 100 ms 串口心跳之间，固件在 `impact_event=true` 时立即额外发送一条运动消息；`impact_count` 用于上位机去重并识别同一危险锁存期间发生的后续撞击。气动状态增加或传递 `automatic_enabled`，使上位机能够准确区分“自动保护未启用”和“自检/压力阻塞”。

## 上位机设计

### 碰撞求助窗口

新增独立的非模态碰撞告警窗口。首次收到新的 `impact_count`，或兼容旧固件时首次观察到 `impact` 上升沿，立即：

- 立即显示并置顶；
- 使用红色危险状态；
- 显示事件时间、加速度模长、`accel_delta_g`、采样间隔；
- 显示气囊状态：已触发、正在充气、保压、已阻塞或故障；
- 显示泵阀自检、压力有效性、自动模式和故障原因；
- 不播放声音。

告警持续显示，直到用户点击“确认已知晓”。关闭求助窗口只清除上位机展示状态，不发送泄压、复位故障或解除固件危险锁存命令。若窗口已显示时再次撞击，更新为最新事件并增加碰撞次数。

### 状态门控

上位机“自动保护允许”状态使用以下共同条件：

```text
automatic_enabled
and pressure_valid
and pressure_age_ms <= 200
and pump_verified
and valve_verified
and not self_test_failed
```

视觉路径另外要求视觉结果新鲜；MPU 撞击路径不依赖视觉新鲜度。界面不得因视觉离线而把已通过自检的 MPU 碰撞保护显示为禁止，也不得在泵阀尚未验证时显示允许。

### 事件记录

`SessionRecorder` 增加碰撞事件记录，保存到当前会话的 `collision_events.jsonl`。每条记录至少包含：

- 主机记录时间；
- 设备时间戳和序号；
- 加速度模长、变化量和采样间隔；
- 相对倾角；
- 气动自动模式、自检和压力状态；
- 气囊实际状态、触发源和故障；
- 用户确认时间；
- 同一求助窗口期间的累计撞击次数。

该文件使用追加式生命周期记录：检测时写入 `detected`，气动状态变化时写入 `pneumatic_update`，用户确认时写入 `acknowledged`，三类记录通过碰撞事件编号关联。这样不会在已有 JSONL 行上做原地修改，也不会在气囊结果尚未返回时伪造最终状态。原始串口行仍按现有机制保存，独立碰撞记录用于快速审计。

## 错误处理

- MPU 未校准：不进行撞击判定，也不触发气囊或求助窗口。
- 时间戳回退、间隔为 0 或超过 20 ms：忽略本次撞击判断并刷新基线。
- 碰撞发生但气动未准备好：保持求助窗口并显示阻塞原因，不伪造充气成功。
- 压力在充气期间失效或过期：沿用固件故障泄压路径，上位机显示实际故障。
- 上位机重启：历史碰撞仍保存在会话文件；不根据旧串口行重新触发弹窗。
- 重复运动心跳：新版固件按 `impact_count` 去重；旧固件只按 `impact` 上升沿兼容处理，不为每条锁存心跳重复创建窗口或日志事件。

## 测试设计

### 固件宿主测试

1. 普通 0.1–0.6 g 模长变化不触发撞击。
2. 10 ms 内 1.19 g 变化不触发，1.20 g 变化触发。
3. 20 ms 内达到阈值触发，21 ms 不触发并刷新基线。
4. 首次撞击后的 200 ms 内即使再次超过阈值也不重复计数，时间窗结束后的新撞击正常计数。
5. 删除旧绝对阈值语义：稳定保持 2.1 g 不因“连续两次”单独触发；只有到达 2.1 g 时产生的模长突变可触发一次。
6. 撞击后在 90° 横向静止 5 秒仍保持锁存。
7. 回到相对基准姿态并稳定 5 秒后解除锁存。
8. 快速倾斜仍产生诊断，但不再成为气动触发源。
9. 撞击在自动模式、自检和压力均有效时进入充气；任一门控失败时不充气。

### 上位机测试

1. 解析带和不带新诊断字段的 motion version 2 消息。
2. 新撞击弹出一个持续求助窗口，普通运动不弹出。
3. 撞击样本绕过 100 ms 心跳周期立即发送，且携带递增的 `impact_count`。
4. 重复锁存心跳不重复创建事件；第二次真实撞击更新计数和详情。
5. 用户确认只关闭上位机求助，不发送气动命令。
6. 自检未通过、压力过期、自动模式关闭分别显示正确阻塞原因。
7. MPU 撞击保护允许状态不依赖视觉新鲜度。
8. `collision_events.jsonl` 包含检测、气动反馈和用户确认记录。
9. 整个碰撞路径不发起 voice/DFPlayer 请求。

- `scripts/verify.ps1`；

- 使用 ESP-IDF 5.4.4 和实际 `sdkconfig.runtime` 做干净构建；

- 不执行 `flash`。

## 验收标准

1. 唯一撞击判据为 `Δ|a| >= 1.2 g` 且 `0 < Δt <= 20 ms`；200 ms 不应期只合并同一次撞击的重复峰值。
2. 普通骑行变化不会报警、求助或充气。
3. 撞击级突变在安全门控满足时触发实际气动策略，不依赖视觉新鲜度。
4. 横向倒地不会自动解除危险锁存。
5. 上位机持续显示求助和真实气囊状态，直到人工确认。
6. 不产生碰撞语音请求。
7. 所有宿主测试、Python 测试、静态检查和 ESP-IDF 构建通过。
8. 文档明确区分软件实现、构建验证与尚未完成的实物气囊验证。

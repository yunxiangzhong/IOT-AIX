# 当前仓库阈值与临界判断汇报

检查时间：2026-07-23  
检查对象：`ProjectFile` 工作区当前源码与本地配置（未刷机、未做物理执行验证）

## 先给结论

系统的视觉风险分界固定为 **30 / 60 / 80 分**：低于 30 为安全、30 起提示、60 起高风险、80 起紧急。灯光、语音和气动的触发条件并不完全相同：

| 风险分数 | 视觉状态 | 本地 RGB | 语音曲目 | 自动气动的候选触发 |
|---:|---|---|---:|---|
| 0–29 | `safe` | 绿灯常亮 | 不播 | 否 |
| 30–59 | `attention` | 黄灯 1 Hz 慢闪 | 1 | 否 |
| 60–79 | `high` | 橙灯 2 Hz 快闪 | 2 | 是，但还须自动气动总开关、闭环压力与新鲜度条件全部满足 |
| 80–100 | `critical` | 红灯双脉冲 | 3 | 是，但还须自动气动总开关、闭环压力与新鲜度条件全部满足 |

最重要的配置判断：仓库存在三个配置层，结论不能混用。

| 配置层 | 气动控制 | 自动气动 | 含义 |
|---|---|---|---|
| `AIX/sdkconfig.defaults` | 关 | 关 | 受版本管理的安全默认值 |
| `AIX/sdkconfig` | 关 | 未配置为开（代码回退为关） | 当前本地生成配置；这是构建通常直接读取的文件 |
| `AIX/sdkconfig.runtime` | 开 | 开 | 运行时覆盖模板；必须经同步、重新构建并刷入才可能成为设备实际行为 |

因此，**当前仓库不能直接得出“设备已经会自动充气”**。如果使用当前 `sdkconfig` 构建，气动控制关闭；如果先将 runtime 覆盖同步到 sdkconfig、再构建刷机，则自动气动会被编译打开，仍要通过下文的压力与风险联锁。

## 1. 视觉风险如何跨过阈值

### 1.1 分数与分级

- 视觉服务和 ESP32 的分段一致：`<30` 低、`>=30` attention、`>=60` high、`>=80` critical。
- YOLO 只保留人员、自行车、汽车、摩托、公交、卡车、交通灯、停车标志；置信度低于 **0.35** 的检测会被过滤。
- 场景接近度本身最高只给 **45 分**，所以“没有目标检测、只有深度场景接近”最多是 attention，不会单独变为 high/critical。
- 有目标时，原始风险 = `100 × (0.40×接近度 + 0.25×画面占用 + 0.20×中心度 + 0.15×检测置信度)`。
- 极端紧急例外同时满足：原始目标分数 `>=92`、相对深度 `<=0.12`、检测置信度 `>=0.85`、目标中心度 `>=0.70`。此时直接把风险推到 critical；否则单目标分数封顶 **79**，不会仅凭单帧进入 critical。

### 1.2 防抖与升降级门槛

- 上升 EMA 系数 **0.45**，下降 EMA 系数 **0.20**：下降更慢，避免画面抖动时刚进入高风险就立刻清除。
- 普通升级要求同一候选分级连续 **2 帧**；降级要求连续 **3 帧**。
- 极端紧急不等待 2 帧，立即 critical。
- 对外发布的分数会被钳制在已经确认的分级内。例如 EMA 已变为 70、但 high 尚未连续确认时，仍不会发布为 high。

## 2. 什么情况亮什么灯

### 本地视觉链路

- 启动后尚未收到有效风险：前 **45 秒**为 `loading`，蓝灯 1 Hz 慢闪。
- 收到的风险帧采集时间距设备当前时间超过 **3 秒**，或最近一次有效风险接收已达到 **3 秒**：紫灯 1 Hz 慢闪（`fault/stale`）。
- 设备 ID、boot ID 不匹配、帧号不递增、风险分数/分级不对应、无效包，同样不会形成新的执行结论。

### 路侧协同告警的仲裁

- 路侧告警只接收 `attention/high/critical`，TTL 必须是 **1–30,000 ms**，且 `simulated=true` 会被拒绝。
- 最高优先级取路侧有效告警与本地视觉状态比较后的“严格更高者”。相同等级不覆盖本地；critical=5、high=4、fault=3、attention=2、safe/loading=1。
- 所以：远端 high 能覆盖本地 attention/fault，但不能覆盖本地 high 或 critical；远端 critical 能覆盖 local high/fault，但不能覆盖本地 critical。
- 路侧 ACK 只证明事件被策略接收和 RGB 仲裁结果；不等于 DFPlayer 已实际播放，也不等于泵/阀已动作。

## 3. 什么情况播语音

- attention/high/critical 分别映射曲目 **1/2/3**；low 不发语音。
- 首次进入任一可播等级立即请求；等级上升立即请求；同等级最多每 **10 秒**重播一次。
- 等级降低不会抢断正在播放的更高级语音；降级后的重复播报需重新经历完整 10 秒冷却。
- 设备端当前播放的曲目编号更高时，低优先级请求被抑制；相同 `command_id` 作为重复请求被幂等确认。DFPlayer 初始化等 TF 卡最多 **3 秒**，失败后每 **3 秒**重试。
- 语音实际成功必须看到设备 `voice_status=playing/finished`，而不是只看风险分数、ACK 或 UI。

## 4. 气动：真正决定是否充气的阈值和联锁

### 4.1 配置和压力阈值

| 项目 | 当前默认/限制 | 行为 |
|---|---:|---|
| 默认目标压力 | 8.0 kPa | 自动充气到此值即进入保压 |
| 默认软件压力上限 | 12.0 kPa | 超过即 `pressure_over_max` 故障泄压 |
| 允许保存的目标压力 | 6.0–20.0 kPa | 超出拒绝保存 |
| 允许保存的上限 | `目标压力`–20.0 kPa | 上限低于目标或超过 20 拒绝 |
| 压力新鲜度 | <=200 ms | 超过视为 stale；连续无效/陈旧 100 ms 后故障泄压 |
| 手动标定单次脉冲 | 200 ms | 仅标定模式 |
| 标定累计泵开启上限 | 2,000 ms | 仅标定模式 |
| 标定压力天花板 | 20.0 kPa | 达到即停止充气 |
| 自动保压最大时长 | 15,000 ms | 到时进入 `hold_timeout` 故障泄压 |
| 自动压力回补差 | 0.3 kPa | 低于活动目标 0.3 kPa 后再次预开阀、充气 |
| 自动释放判据 | 目标压力 - 1.0 kPa 或泄压满 5 秒 | 进入 1 秒冷却 |
| 完全泄压判据（手动/标定） | <=0.5 kPa 或泄压满 5 秒 | 进入 1 秒冷却 |

另有压力传感器诊断阈值：ADC 电压 **100–2900 mV** 才是原始有效，200–2700 mV 线性映射为 0–200 kPa；滤波压力到 **180 kPa** 仅记录传感器软过压警告。真正控制泵阀的是上表的 `max_kpa`（默认 12、硬限制 20），不是 180 kPa。

### 4.2 自动充气的必要条件

自动充气不是“风险 >=60 就一定执行”。必须同时满足：

1. 构建/运行配置把自动气动打开；
2. 标定配置有效（目标和上限通过范围校验）；
3. 压力样本存在、有效且年龄不超过 200 ms；
4. 视觉决策仍有效且未 stale；
5. 风险状态为 high 或 critical，或 MPU6050 已校准且识别为冲击；
6. 没有急停、压力故障、超上限等故障。

到达触发条件后，阀先开 **20 ms**，再开泵充气。到达目标压力即停泵并保压；没有“自动充气满 60 秒强制停止”这一旧规则。风险消失后必须持续 **5 秒**确认无触发，才会转泄压。

### 4.3 强制停泵/泄压优先级

以下任一项优先于继续充气：

- 急停：立刻进入故障泄压；
- 压力无效或陈旧：先容忍最多 100 ms，之后故障泄压；
- 压力 `> max_kpa`：立即故障泄压；
- 收到手动泄压：直接进入泄压；
- 自动期间，风险包明确带有 `actuation_hazard_active=false`：快速泄压；
- 自动保压超过 15 秒：故障泄压。

注意：气动源码虽然仍有自检相关常量（0.3 kPa 起压、800 ms 泵时长、6 秒超时），但 `SELF_TEST` 命令目前显式返回 `self_test_disabled_pressure_feedback_mode`，不能把这套未启用自检当作自动气动允许条件。

## 5. MPU6050 的临界判断

| 项目 | 阈值 |
|---|---:|
| 开机标定 | 连续 200 个样本，且每个样本加速度 0.9–1.1 g、角速度 <10 °/s |
| 冲击 | 相邻样本加速度模长变化 >=1.2 g，采样间隔 <=20 ms |
| 冲击不应期 | 200 ms |
| 快速倾斜 | 倾角 >45° 且角速度 >=80 °/s，持续 >=200 ms |
| 危险锁存清除 | 倾角 <30°、加速度 0.8–1.2 g、角速度 <20 °/s，持续 >=5 秒 |

代码当前实际自动气动触发函数使用的是“视觉 high/critical 或 MPU 冲击”。`rapid_tilt` 会被采集、上报、锁存，但在这一路自动触发函数里没有被纳入触发条件；不要把“快速倾斜=true”解释成必然自动充气。

## 6. 演示模式的边界

- 演示会话 lease 允许 **5–60 秒**，服务端默认 15 秒；心跳 5 秒一次。
- 演示场景仅允许 4/5/6，且它们将 RGB 强制为红双脉冲/橙 2 Hz/紫 1 Hz，并以 95 分的 critical 演示决策呈现。
- 演示动作接口还要求气动状态报告“自动已开启且无故障”；但响应中的 `pneumatic.requested=true` 是请求/控制模式字段，不能替代泵阀状态与压力闭环遥测。

## 7. 建议作为验收清单的关键观察点

1. 构建前先固定使用的是 `sdkconfig` 还是 `sdkconfig.runtime` 同步后的配置；这是自动气动是否可能开启的第一道判定。
2. 观察 `action_status` 的 `risk_score / action_state / stale / rgb_pattern`，确认 30、60、80 和 3 秒失效是否符合预期。
3. 语音以 `voice_status` 为准；气动以 `pneumatic_status` 中的 `state / fault / pump_on / valve_on / pressure_kpa / pressure_age_ms` 为准。
4. 自动气动试验前，必须实测压力闭环、急停、失压/陈旧、超上限、风险消失 5 秒、保压 15 秒故障泄压；当前仅完成源码与配置审阅，本报告不构成物理安全验收。

# AIX 脉盔项目审阅与优化建议

> 审阅日期：2026-07-10  
> 审阅范围：`ProjectFile/` 当前代码、目录结构、三份 README，以及本机可见测试。  
> 本报告只描述当前仓库真实状态，不把规划中的摄像头检测、语音硬件和泵阀控制写成已完成。

## 一、项目理解与整体实现建议

### 1. 项目定位

AIX 脉盔是一套面向电动车和摩托车骑行场景的智能安全头盔原型。它希望把环境感知、风险判断、骑手提醒和气囊执行连接成闭环：

```text
压力 / 视觉感知
      ↓
ESP32-S3 风险判断
      ↓
语音提醒 + 分级气囊目标
      ↓
气泵 / 电磁阀 / 气囊执行
      ↓
压力反馈与失效保护
```

当前仓库已经搭起了闭环的接口骨架，但还不是完整产品：真实压力采集已进入固件；目标检测、语音播放和泵阀动作仍分别由模拟检测事件、串口事件和模拟执行事件代替。

### 2. 当前实际运行链路

```text
链路 A：真实压力
XGZP6847A -> ESP32-S3 ADC -> pressure NDJSON -> PC 上位机曲线/状态

链路 B：默认演示闭环
ESP32-S3 模拟 vision_detect -> risk_fusion v2
-> risk / voice / actuator NDJSON -> PC 上位机展示

链路 C：临时摄像头输入
PC/手机摄像头 -> OpenCV 光流趋势 -> vision v1 -> ESP32-S3
-> risk_fusion v1 fallback -> risk / actuator NDJSON
```

Demo 配置会启动模拟 `vision_detect` 任务，因此运行一段时间后通常优先走链路 B；Hardware 配置会关闭该任务。PC 摄像头链路已经接通，但当前主要是 fallback 和接口验证，并非真实目标检测主链路。

### 3. 已有实现的优点

- 固件、上位机和串口事件边界基本清楚，适合分模块联调。
- 压力、视觉、风险、语音、执行分别有独立数据结构或模块，没有全部堆在入口函数中。
- 真实硬件、模拟输入和上位机展示之间的完成状态区分较诚实。
- `vision` v1 与 `vision_detect` v1 并存，为临时 PC 摄像头和未来端侧检测保留了迁移路径。
- 上位机已有线程化串口/摄像头读取、事件路由、曲线和基础测试，作为比赛调试台是合适的。

### 4. 整体实现建议

#### P0：已完成的工程基线

1. **测试可复现。** 已停止忽略 `AIX/test/` 和 `host_app/tests/` 的测试源码；新增 `scripts/verify.ps1` 统一运行 Python、`compileall` 和 7 组固件主机侧 C 测试，产物写入被忽略的 `.test-bin/`。
2. **Demo/Hardware 配置。** 已通过 `Kconfig.projbuild`、`sdkconfig.defaults` 与 `sdkconfig.hardware.defaults` 控制模拟视觉；Demo 默认开，Hardware 默认关。
3. **v2 新鲜度保护。** `vision_detect` snapshot 记录本机接收时间；达到 `CONFIG_AIX_VISION_DETECT_STALE_MS` 后回退到 v1，两个输入都不可用时输出安全 0%。
4. **正常风险语义。** 无目标和远目标均为 `level=0、target_pct=0`；预充气若需要，应在未来安全状态机中作为独立状态实现。

#### P1：接入真实视觉，但保持接口可替换

建议继续保留 ESP32-S3 作为主控制器，第一版采用 `OV5640 + PSRAM + 低分辨率/低帧率轻量检测`。模块按下面的边界推进：

```text
camera_source
    -> detector
    -> distance_estimator
    -> vision_detect snapshot
    -> risk_policy
```

- `camera_source` 只负责初始化、采帧、帧状态和错误恢复。
- `detector` 只输出类别、置信度和 bbox，不直接控制气囊。
- `distance_estimator` 根据 bbox、类别宽度假设和标定参数输出粗距离与接近趋势。
- `risk_policy` 只消费规范化感知结果，不依赖具体摄像头或模型。
- 初期只检测车辆、行人、障碍物少量类别；优先证明稳定闭环，不追求高分辨率和多类别。

#### P1：执行层先做安全状态机，再接真实泵阀

真实执行层不应继续沿用“风险百分比直接等于充气百分比”的简单映射。建议至少定义：

```text
DISARMED -> READY -> PREINFLATE -> PROTECT -> HOLD -> VENT -> FAULT
```

转换条件应同时考虑压力有效性、过压、视觉新鲜度、通信状态、人工急停、动作超时和泄气能力。硬件层再通过 `actuator_hal` 封装 GPIO、PWM、MOS 管和阀，避免风险算法直接操作引脚。

#### P2：建立可量化的比赛演示证据

- 把 CSV 从压力记录扩展为统一事件日志，记录 `pressure / vision_detect / risk / voice / actuator / fault`。
- 记录端到端延迟、摄像头帧率、检测耗时、误报/漏报、压力响应时间和过压停止时间。
- 准备 `demo` 与 `hardware` 两套配置，让答辩时可以明确区分模拟闭环和真实硬件闭环。
- 单目距离只表述为“粗估计”，通过固定距离标定表给出误差范围，不包装成精确测距。

## 二、代码结构优化建议

### 1. 当前结构判断

当前目录对原型阶段是够用的，问题不在于文件数量少，而在于部分文件开始同时承担多种职责：

- `AIX/main/risk_fusion.c` 同时包含风险纯函数、FreeRTOS 任务、数据源选择、NDJSON 输出、语音策略和执行调用。
- `host_app/aix_host_app/app.py` 同时负责 UI 组装、串口生命周期、事件路由、摄像头生命周期、模拟数据和 CSV 记录。
- C 与 Python 分别维护协议解析，没有一份独立、可版本化的协议规范。
- 参数散落在多个 `.c` 文件的宏中，难以形成 demo/hardware 配置。

### 2. 建议的演进结构

不要立刻为了“整齐”做大搬家；建议在接入真实摄像头和泵阀时，按职责逐步拆分：

```text
ProjectFile/
├─ AIX/
│  ├─ main/                    # 只保留 app_main 与系统装配
│  └─ components/
│     ├─ sensing/              # pressure_sensor、camera_source
│     ├─ perception/           # detector、distance_estimator、vision snapshot
│     ├─ decision/             # risk_policy、safety_state_machine
│     ├─ actuation/            # voice_prompt、actuator_hal、airbag_controller
│     └─ transport/            # NDJSON 输入输出、config、事件编码
├─ host_app/
│  └─ aix_host_app/
│     ├─ controllers/          # serial、camera、recording 生命周期
│     ├─ protocol/             # models、parsers、encoders
│     ├─ services/             # history、simulation、event recorder
│     ├─ vision/               # PC 临时视觉输入
│     └─ widgets/              # 只负责展示和用户输入
├─ docs/
│  ├─ architecture.md
│  ├─ protocol/ndjson.md
│  └─ 项目审阅与优化建议.md
├─ scripts/
│  └─ verify.ps1
└─ README.md
```

### 3. 具体拆分顺序

1. **先拆固件风险模块。** 保留 `risk_fusion_evaluate*()` 为纯函数；把 FreeRTOS 循环移到 `risk_task.c`，把 JSON 输出移到 `event_encoder.c`，把语音文案选择移到 `voice_policy.c`。
2. **再拆上位机主窗口。** `app.py` 只装配界面和控制器；串口、摄像头、记录分别由 `SerialController`、`CameraController`、`EventRecorder` 管理。
3. **集中协议定义。** 新增协议文档，明确每种事件的版本、必填字段、单位、范围、发送方向和兼容策略。C 端暂时不必引入大型 JSON 库，但应增加边界测试。
4. **集中配置。** 把周期、阈值、是否启动模拟输入、预充气比例等转成 Kconfig 或配置结构；硬件引脚与算法阈值分开。
5. **完善多目标语义。** `vision_detect_result_t` 支持 8 个对象，但当前 C 解析器只提取第一个对象，`nearest_class` 也直接取 `objects[0]`。真实检测接入前，应明确对象排序约定或按距离寻找最近目标。
6. **保持 UI 薄层。** widget 只渲染模型和发出用户意图，不直接承担串口协议、摄像头算法或文件写入。

### 4. 不建议现在做的重构

- 不建议立即更换 ESP32-S3 主控或重写为另一套框架。
- 不建议为了目录层级一次性移动全部文件；应随真实模块接入逐步迁移。
- 不建议在目标检测尚未跑通前引入云端平台、复杂消息队列或数据库。
- 不建议第一版同时做双目、IMU、雷达和多模型融合，容易失去主线。
